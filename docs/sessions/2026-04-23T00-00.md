## 3. Decisions Made

| Decision | Rationale |
|----------|-----------|
| `Security` name is good | "Security" is the established finance term for any tradeable instrument |
| Keep `SecurityCriteria` name | Functional and unambiguous; `SecurityQuery` was noted as an alternative but not adopted |
| Keep `target_price` and `target_date` as separate fields | Each can be useful independently: `target_date` alone for point-in-time resolution; `DataSource.validate()` enforces both when needed |
| FIGI deserves a full VO with checksum validation | FIGI has a well-defined 12-char format and Luhn check digit — same justification as ISIN |
| FIGI VO mirrors ISIN pattern exactly | Consistency with the existing Namespace Pattern across the codebase |
| Test FIGI validation via `validate_figi()` directly, not `FIGI(...)` | `RootModel.__init__` bypasses `BeforeValidator`; the validator only runs when used as a Pydantic field type |

---

## 4. Decisions Refused / Alternatives Rejected

| Rejected Approach | Reason |
|-------------------|--------|
| `PriceOnDate` nested model combining `target_price` + `target_date` | Both fields are independently useful in `resolve()` and `validate()` already enforces both at its own signature level |
| Adding `conid`, `yahoo_symbol`, `ft_id` to `Security` now | Discussed but not implemented this session; deferred |
| Adding cross-field validator requiring `target_date` when `target_price` is set | Fields are legitimately used independently |
